# 无人机系统

## 一、无人机各模块通信

在了解无人机各模块通信之前首先要理解两个概念

- **MAVROS**本质上是一个连接MAVLink通信桥梁的ROS节点，它的核心作用是将 MAVLink 协议的消息转化为 ROS 开发者熟悉的 **ROS Topics** 和 **Services**。使得感知和决策端与飞控可以进行通信

- **MAVLink**是一个通信协议，它定义了无人机与地面站以及无人机内部组件之间数据的交换法则。

### 1、SLAM和飞控

**PX4飞控**接收**SLAM**发送的位姿信息，主要涉及两个话题`/mavros/vision_pose/pose`和`/mavros/odometry/out` 

- `/mavros/vision_pose/pose` 是经典接口，主要为飞控提供位姿(位置和姿态)信息

- **VINS** 算法不仅能输出**位姿**还能输出**速度和协方差**，因此向飞控发送`/mavros/odometry/out`话题更好

| **特性**           | **/mavros/vision_pose/pose (Pose)**                          | **/mavros/odometry/out (Odometry)**                          |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **消息类型**       | `geometry_msgs/PoseStamped`                                  | `nav_msgs/Odometry`                                          |
| **飞控获取的信息** | 位姿                                                         | 位姿和速度                                                   |
| **速度估算方式**   | **一阶微分：** 飞控通过前后两帧位置差来算速度，延迟高且噪声大 | **直接观测：** 飞控直接得到 VINS 计算的瞬时速度，响应极快    |
| **动态表现**       | 由于速度估计不准，急停或高速飞行时，飞机容易产生晃动         | 飞行丝滑，能支撑更激进的路径规划轨迹                         |
| **容错性**         | 如果视觉丢帧，飞控速度估计会瞬间出错。                       | 即使视觉短暂丢帧，VINS 输出的惯性速度仍能支撑飞控短时间维持平稳。 |

SLAM发送给飞控的位姿会参与**EKF2的融合**，将融合后得到的位置、速度、姿态



### 2、SLAM和规划

SLAM只需要发送位姿给路径规划模块即可，路径规划只需要得到位姿来确定路径点，不需要速度等信息，接收位姿频率要求较飞控低，具体的问题具体分析

| **比较维度**   | **发送给路径规划 (Planning)**                  | **发送给飞控 (EKF2 融合)**                                   |
| -------------- | ---------------------------------------------- | ------------------------------------------------------------ |
| **数据身份**   | **当前位置反馈**                               | **外部观测值**                                               |
| **用途**       | 用于计算与目标点的“偏差”，从而决定下一步飞向哪 | 用于和 IMU 预测值进行“对齐”，消除 IMU 的累积误差             |
| **时间戳要求** | 只要不太旧即可（30ms 延迟通常可接受）。        | **极其严苛**。必须包含精确的 Hardware Sync 时间戳，否则 EKF2 无法对齐 IMU 序列。 |
| **频率**       | 只要匹配规划频率（如 20Hz-50Hz）。             | 越快越好，且必须稳定，频率剧烈跳动会导致飞控切回手动模式。   |



### 3、路径规划和飞控

在无人机自主飞行系统中，路径规划模块（规划层）主要通过 **MAVROS** 向 **飞控（执行层）** 发布控制指令。

#### 发布的核心话题

最常用且功能最强大的话题是：

`/mavros/setpoint_raw/local`

- **消息类型**：`mavros_msgs/PositionTarget`
- **特点**：它允许同时发送**位置 (Position)**、**速度 (Velocity)** 和 **加速度 (Acceleration)** 指令，并且可以通过一个“掩码” (`type_mask`) 灵活选择生效的参数。

其他常用话题还包括：

- **`/mavros/setpoint_position/local`**：仅发送目标位置（`PoseStamped`）。
- **`/mavros/setpoint_velocity/cmd_vel`**：仅发送目标速度（`TwistStamped`）。

------

#### 话题的具体作用

路径规划模块发布这些话题，主要起到以下三个关键作用：

1. **定义“期望状态” (Target State)** 飞控内部运行着位置、速度和姿态三个闭环 PID 控制器。路径规划模块的作用是为这些控制器提供**输入参考值**（也就是告诉飞控：“我希望你下一时刻出现在这里，并保持这个速度”）。
2. **维持 Offboard 模式控制** PX4 进入 Offboard 模式有一个前提：必须以 **> 2Hz** 的频率持续接收来自外部（如你的路径规划模块）的 setpoint 指令。如果指令中断，飞控会因为安全机制自动切出 Offboard 模式。
3. **平滑轨迹的物理实现** 如果你使用的是 `/mavros/setpoint_raw/local` 并加入了速度或加速度**前馈**，飞控能预先知道运动趋势，从而让无人机在跟踪 SLAM 规划出的复杂曲线轨迹时，飞行表现得更加丝滑，减少滞后和震荡。

------

#### 数据流走向图

- **路径规划器**：根据地图和 VINS 给出的位姿，计算出下一个航点。
- **MAVROS**：将 ROS 格式的指令打包成 **MAVLink** 协议。
- **PX4 飞控**：接收到指令后，对比当前 SLAM 给出的实际位姿，计算电机出力，完成动作。







### 无人机自主飞行的完整数据闭环 (从感知到动作)：

您可以沿着图中的箭头，清晰地看到数据是如何在各个 drone-specific 模块之间流动的：

1. **感知与状态递推 (高频IMU)**:
   - **Gazebo** 生成高频的 **IMU数据**。
   - **SLAM算法** 实时订阅 `/iris_0/mavros/imu/data` 话题。由于 IMU 频率极高（~200Hz），SLAM 的后端可以利用它进行快速的状态递推，维持短时间内的相对定位。
2. **图像配准与零偏矫正 (低频相机)**:
   - **Gazebo** 生成低频（~30Hz）的 **相机图像**（ `/iris_0/stereo_camera/left/image_raw` 等）。
   - **SLAM算法** 订阅该话题。SLAM 的前端利用图像提取点线特征，通过帧间匹配和后端优化，来对 IMU 的累积漂移和零偏（Bias）进行矫正，从而提供长期稳定的绝对定位。
3. **位姿反馈与 EKF 融合**:
   - **SLAM算法** 将解算出的 **视觉/外部位姿** 发布到 `/mavros/vision_pose/pose` 话题。
   - **MAVROS** 转发该数据给 **PX4 EKF2**。
   - **PX4** 将外部视觉位姿与自身的 IMU、气压计、磁力计等数据进行深度的融合，生成无人机最终的位置、速度和姿态估计，并发布给上层使用。
4. **自主路径规划**:
   - **规划与决策** 模块同时订阅 **SLAM输出的估计位姿** 和 **PX4反馈的飞行状态**。
   - 根据预设的 **目标点指令** (Waypoints)，路径规划器计算出下一时刻最安全的期望位置或速度。
5. **控制下发与底层执行**:
   - **规划与决策** 模块将期望速度发送给 **MAVROS**。
   - **MAVROS** 翻译成 **MAVLink控制指令** 发送给 **PX4** 的 Offboard 模式控制器。
   - **PX4** 计算四个电机的 PWM 混控信号。
   - PWM 信号作为 **物理仿真数据** 传给 **Gazebo**。
   - **Gazebo** 中的无人机模型产生物理动作，移动到新的位置，从而开启下一个感知循环。



## 二、坐标变换关系

由于无人机系统中VINS、mavros和px4的参考坐标系不同，因此当飞控在接收VINS发来的位姿时需要进行坐标变换。

## 1、里程计的坐标变换关系

### 1.1 world坐标系

里程计一般都会有一个world坐标系作为参考系，然后估计body坐标系在world坐标系的位姿。

以VINS-Fusion为例

VINS-Fusion中主要包含：world坐标系、body坐标系、camera坐标系，其中的body坐标系也就是IMU的坐标系，camera坐标系是相机的坐标系(双目则用左目作为相机坐标系)，而world坐标系是在初始化完成后通过关键帧选取和重力对齐两个方式结合产生的。

（1）选取出是滑窗中第一帧的body坐标系位置作为world坐标系的位置

`Estimator::visualInitialAlign`函数中的下面的代码可以说明：

```c++
// 下面开始把所有的状态对齐到第0帧的imu坐标系
for (int i = frame_count; i >= 0; i--)
    // twi - tw0 = toi,就是把所有的平移对齐到滑窗中的第0帧
    Ps[i] = s * Ps[i] - Rs[i] * TIC[0] - (s * Ps[0] - Rs[0] * TIC[0]);
```

（2）将world坐标系与重力方向对齐

```c++
// 所有的P V Q全部对齐到第0帧的，同时和对齐到重力方向
Matrix3d R0 = Utility::g2R(g);  // g是枢纽帧下的重力方向，得到R_w_j
double yaw = Utility::R2ypr(R0 * Rs[0]).x();    // Rs[0]实际上是R_j_0
R0 = Utility::ypr2R(Eigen::Vector3d{-yaw, 0, 0}) * R0;  // 第一帧yaw赋0
g = R0 * g;
//Matrix3d rot_diff = R0 * Rs[0].transpose();
Matrix3d rot_diff = R0;
for (int i = 0; i <= frame_count; i++)
{
    Ps[i] = rot_diff * Ps[i];
    Rs[i] = rot_diff * Rs[i];   // 全部对齐到重力下，同时yaw角对齐到第一帧
    Vs[i] = rot_diff * Vs[i];
}
```

IMU测的实际上是重力的平衡力，而VINS-Fusion中的g是正的，因此直接把IMU测的支持力和正的g对齐即可，此时他们都是朝向z轴正方向！！！

此外，由于仅仅通过重力对齐，得到的 world 坐标系并不是唯一的，此时 world 坐标系绕 z 轴旋转所产生的坐标系 都可以保证重力对齐，因此为了**保证唯一性**，VINS-Fusion 将 world 坐标系的 x 轴方向与初始滑窗中第一帧 IMU 的 x 轴进行了对齐

从而可以得到一个新的坐标系，这两个坐标系之间会有一个旋转关系，这个旋转关系可以通过一次重力对齐和x轴对齐旋转获得！



VINS-Fusion中world坐标系的确立方式为：

在VISN-Fusion初始化完成后，以相机的初始位置作为world坐标系的位置，以重力方向作为z轴，相机x方向在水平面上的投影作为world坐标系的x轴，最后根据右手准则确定y轴！！！

## 2、Mavros坐标变换关系

### 2.1 mavros坐标系介绍

mavros 的坐标变换关系主要分为有三种：东北天坐标系（ENU），北东地坐标系（NED），前左上坐标系（FLU）

以 D435i 相机和 Pixhawk 6C飞控为工具，尝试缕清 mavros 中的坐标系与VINS-Fusion里程计中的坐标系之间的关系。

由于 mavros 中常用的坐标系有**global 坐标系，local坐标系，body坐标系**，其中 global 坐标系指的是经纬度坐标，常在使用 GPS 时才会产生（应用较少），因此主要关注 local 坐标系和 body 坐标系

首先启动mavros节点，终端输入：

```shell
roslaunch mavros px4.launch
```

此时我们查看对应于机体在local坐标系下的位姿的 `/mavros/local_posion/pose` 话题，终端输入：

```shell
rostopic echo /mavros/local_position/pose
```

当没有里程计输入时，上面的输出为空，mavros 并没有自己的局部坐标信息，这意味着目前飞控并没有产生局部坐标系，所以网上所谓的 “局部坐标系以指的是机体上电时的位置作为原点” 的言论是不正确的，因为此时还没有产生局部坐标系！（事实上，这是里程计的效果，飞控并没有生成里程计信息的能力，只有 IMU 信息）

### 2.2 VINS-Fusion到Mavros的坐标变换

接下来将 VINS-Fusion 的里程计输出发布给 mavros 的 /mavros/vision_pose/pose 话题，但是这里会涉及到一个问题：VINS-Fusion 的 world 坐标系 与 mavros 中的 map 坐标系（发布给 /mavros/vision_pose/pose 话题的位姿就是其在 map 坐标系中的位姿，即local 坐标系）并不是对齐的，因此不能直接地将 VINS-Fusion 中里程计的位姿直接发布给机体，而是需要做一定的坐标变换！

mavros的**map坐标系原点是 mavros 收到的里程计信息时飞控所在的位置**，并且其 map 坐标系的 x 轴朝东，y 轴朝北，z 轴朝天，也就是所谓的东北天坐标系（ENU），此时机体方向朝东，也就是 x 轴正向！
